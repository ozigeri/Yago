Funkcionális specifikáció

Projekt neve: Yago Environment Builder

Dátum: 2025.11.03

Készítette: Lénárt János, Ozibiusz Gergő, Köhler Gergő

1. Bevezetés

A Yago Environment Builder egy Windows Forms alapú asztali alkalmazás, amely a fejlesztők munkáját segíti azáltal, hogy automatizálja és leegyszerűsíti a különböző fejlesztői környezetek létrehozását.

A cél, hogy a felhasználók könnyedén összeállíthassák a kívánt környezetet komponensekből (PHP, Node.js, Composer), sablonokat készíthessenek, és új repositorykat generálhassanak a beállítások alapján.

A specifikáció célja, hogy pontosan meghatározza a rendszer funkcionális és nem funkcionális követelményeit, az adatstruktúrákat, folyamatokat, felhasználói felületeket, tesztelési és hibakezelési elveket, ezzel támogatva a fejlesztést, tesztelést és későbbi karbantartást.

1.1. Célközönség

  • Elsősorban fejlesztők, akik gyakran dolgoznak több projekt párhuzamos kezelésével, és igénylik a gyors, automatizált környezetbeállítást.
  • Fejlesztők, akik a Yago alkalmazást implementálják vagy továbbfejlesztik.

1.2. Fogalmak

  • Template (sablon): Elmentett konfiguráció (PHP/Composer/Node verziók), amely a Versions modulból felhasználható.
  • Environment start: A kiválasztott verziókkal történő lokális fejlesztői környezet inicializálása.
  • Repository: Fájlstruktúra inicializálással (opcionálisan GitHub remote-tal), a kiválasztott alkalmazás típusának megfelelően létrehozva.

2. Rendszeráttekintés

  • Alkalmazás típusa: Windows Forms asztali alkalmazás
  • Fejlesztési nyelv: C#
  • .NET Framework: 4.7.2
  • Adattárolás: helyi szöveges fájl (Templates.txt), konfigurációs fájlok

2.1. Fő modulok

  • EntryPoint (Kezdőképernyő): navigáció a fő modulokra, logó és információk megjelenítése.
  • Versions modul: környezetkomponensek verzióinak kiválasztása, sablonok mentése és betöltése, környezet indítása.
  • Repository Creator modul: új repository létrehozása a kiválasztott verziók és beállítások alapján.
  • Templates modul: sablonok listázása, szerkesztése, törlése és mentése.

2.2. Logikai architektúra

  • Prezentációs réteg: Windows Forms UI (EntryPoint, Versions, Repository Creator, Templates formok, popupok).
  • Alkalmazás logika: verziókezelés, sablonkezelés, repository generálás, Git integráció, IDE indítás.
  • Adatkezelés: Templates.txt olvasása/írása, konfigurációs fájlok kezelése.
  • Külső integrációk: opcionálisan GitHub (GIT CLI alapú kommunikáció), IDE futtatása.

3. Felhasználói felületek áttekintése

Kezdőképernyő (EntryPoint)

Cél: a fő modulokhoz való gyors navigáció biztosítása.

  • Fő elemek:
    • Yago logó
    • Navigációs gombok/fül: Versions, Repository Creator, Templates
  • Jellemzők: Egyszerű, áttekinthető elrendezés; minden fő modul egy kattintással elérhető.

Versions modul

Cél: a fejlesztői környezet komponenseinek verzióinak kiválasztása és kezelése.

  • Fő elemek:
    • Legördülő listák a komponensekhez: PHP verzió, Composer verzió, Node.js verzió
    • Gombok:
      • Environment Start
      • Save as Template
      • Load Template
      • Version paths... (komponensek elérési útvonalainak beállítása)
    • Template név beviteli mező (mentéshez)
  • Jellemzők: Csak érvényes verziók választhatók; mentéskor a sablon adatai frissülnek a Templates.txt fájlban; hibák esetén visszajelzés jelenik meg.

Repository Creator modul

Cél: új projekt repository létrehozása a felhasználó által megadott beállítások alapján.

  • Adatbeviteli mezők:
    • App típusa (pl. React, Laravel, Angular, Vue, stb. – legördülő lista)
    • Projekt neve (szövegmező)
    • Mentés helye (szövegmező + „Browse...” gomb)
    • IDE választása (legördülő lista: None, VS Code, PhpStorm)
    • Git létrehozása kapcsoló/checkbox
    • Verziók kiválasztása (Versions modulból örökített vagy külön popupban kiválasztott verziók)
  • Gombok:
    • Select Version – verziók kiválasztása popup ablakban
    • Git Auth – GitHub adatok megadása popupban
    • Create Repository – repository létrehozása
  • Jellemzők: Ellenőrzi a projekt név és mentési hely érvényességét; automatikusan létrehozza a repository szerkezetét; sikeres vagy sikertelen létrehozásról visszajelzést ad, opcionálisan pusholja egy új Git repositoryba, opcionálisan megnyitja IDE-ben a repositoryt.

Templates modul

Cél: a sablonok kezelése (listázás, szerkesztés, törlés, mentés).

  • Fő elemek:
    • Táblázatos lista: soronként egy sablon (Template név)
    • Edit gomb: a kiválasztott sablon szerkesztése (popup ablak)
    • Edit popup:PHP verzió, Composer verzió, Node.js verzió
      • Delete Template gomb: jelölt sablon(ok) törlése
    • Checkbox oszlop a sorok mellett a tömeges törléshez
  • Jellemzők: Sablonok könnyen kiválaszthatók és kezelhetők; mentéskor a Templates.txt fájl frissül; hibák esetén visszajelzés jelenik meg.

4. Funkcionális folyamatok

4.1. Versions modul folyamata

  • A felhasználó megnyitja a Versions fület.
  • A rendszer betölti a rendelkezésre álló PHP, Composer és Node.js verziók listáját.
  • A felhasználó kiválasztja a kívánt verziókat a mezőkben.
  • Opcionálisan megad egy Template nevet.
  • A felhasználó megnyomja az Environment Start gombot.
  • A rendszer ellenőrzi:
    • minden kötelező mező ki van-e töltve,
    • a verzióformátum megfelelő-e,
    • az elérési útvonalak érvényesek-e.
  • Ha az ellenőrzés sikeres: a rendszer inicializálja a környezetet (lokális konfigurációk, esetleges script futtatások).
  • A felhasználónak van lehetősége:
    • a template nevét megadni és a Save as Template gombra kattintva elmenteni a megadott konfigurációkat,
    • egy meglévő template beállításait a Load Template gombbal beimportálni.
  • A művelet eredményéről a rendszer vizuális visszajelzést ad (MessageBox).

4.2. Repository Creator modul folyamata

  • A felhasználó megnyitja a Repository Creator fület.
  • A felhasználó megadja az App típusát, a Projekt nevét és a Helyét.
  • Verzió kiválasztása: A Select Version gombra kattintva egy popup ablakban választhatja ki a PHP/Composer/Node verziókat.
  • A felhasználó kiválasztja az IDE-t (vagy None értéket hagy).
  • Opcionálisan bekapcsolja a Git létrehozása kapcsolót.
  • Amennyiben Git létrehozás be van kapcsolva, a felhasználó a Git Auth gombra kattintva megadja:
    • GitHub felhasználónevét,
    • GitHub tokenjét,
  • A Create Repository gombra kattintva a rendszer:
    • ellenőrzi a projekt név és mentési hely érvényességét,
    • ellenőrzi, hogy a mappa nem ütközik létező struktúrával (pl. nem üres könyvtár),
    • létrehozza a repository könyvtárstruktúrát az App típusának megfelelő skeleton alapján,
    • git inicializálást végez (ha be van kapcsolva),
    • GitHub repositoryt hoz létre és pushol (ha be van állítva Git Auth és Git opció).
  • Amennyiben nem None értéket választott ki az IDE megadásánál, a megadott IDE-ben megnyílik a repozitórium (parancssoros hívással).
  • Hiba esetén a folyamat megszakad, részletes hibaüzenet jelenik meg.

4.3. Templates modul folyamata

  • A felhasználó megnyitja a Templates fület.
  • A rendszer beolvassa a Templates.txt fájlt, és táblázatban megjeleníti a sablonokat.
  • Edit Template: Az Edit gombra kattintva popup ablak nyílik, ahol módosíthatók a verziók (és szükség esetén a sablon neve).
  • A Save gomb megnyomása után a rendszer:
    • ellenőrzi a mezők érvényességét,
    • frissíti a Templates.txt fájlt,
    • frissíti a listát a UI-ban.
  • Delete Template: A törölni kívánt sablonok checkbox-ának bejelölése és a Delete Template gomb megnyomása után – megerősítést követően – a rendszer:
    • eltávolítja a kijelölt sablonokat a Templates.txt fájlból,
    • frissíti a listát a felületen.

5. Adatstruktúrák és fájlformátumok

5.1. Templates.txt formátuma

A sablonok egy helyi Templates.txt fájlban kerülnek tárolásra. A fájl soronként egy sablont tartalmaz. Formátum: pontosvesszővel (;) tagolt értékek.

Példa sor:

TemplateName;PHPVersion;ComposerVersion;NodeVersion

Konkrét példa:

LaravelDefault;8.2;2.7;18.0.0

  • TemplateName: egyedi sablonnév (string, nem üres, nem tartalmazhat ; karaktert).
  • PHPVersion: verziószám (pl. 7.4, 8.1, 8.2).
  • ComposerVersion: verziószám (pl. 2.5, 2.7).
  • NodeVersion: verziószám (pl. 16.0.0, 18.0.0).

5.2. Template objektum modell szinten

  • Properties:
    • Name: string
    • PHPVersion: string
    • ComposerVersion: string
    • NodeVersion: string

8. Rendszerkövetelmények

  • A Yago alkalmazás kizárólag Windows operációs rendszeren futtatható.
  • A működéshez szükséges a .NET Framework 4.7.2 verzió telepítése.
  • Az alkalmazás nem tartalmaz automatikus verziófrissítési funkciót, manuális frissítés szükséges.
  • Egyfelhasználós asztali alkalmazás.
  • Minimális hardverkövetelmény: modern, 64 bites Windows gép, legalább 4 GB RAM ajánlott.

9. Nem-funkcionális követelmények

9.1. Teljesítmény

  • A fő alkalmazás ablak (EntryPoint) megnyitása átlagos gépen legfeljebb 2 másodperc alatt történjen.
  • A Templates lista betöltése legfeljebb 1 másodperc legyen 100 sablonig.
  • Egy új repository létrehozása (fájlrendszer műveletek + git init, GitHub push nélkül) átlagos esetben legfeljebb 2 perc legyen.

9.2. Használhatóság

  • Minden kötelező mező mellett legyen egyértelmű címke.
  • Hiba esetén a hibaüzenet egyértelműen jelezze, melyik mező vagy lépés érintett.

9.3. Stabilitás

  • Fájl- és IO-művelet try/catch blokkokkal legyen körülvéve.
  • Hiba esetén az alkalmazás nem záródhat be váratlanul; a felhasználó megfelelő üzenetet kap.

9.4. Biztonság

  • A GitHub tokenek nem kerülnek logolásra semmilyen formában.

9.5. Karbantarthatóság

  • A verziólisták frissíthetősége konfigurációból (nem kódmódosítással) ajánlott.
  • A modulok logikája külön osztályokba legyen szervezve (UI logika vs. üzleti logika szétválasztása).

10. Tesztelési terv és elfogadási kritériumok

10.1. Funkcionális tesztesetek (példák)

  • TS-01: Üres projekt név – Repository Creator modulban üres Projekt neve mező esetén a Create Repository gomb megnyomására hibaüzenet jelenik meg, a repository nem jön létre.
  • TS-02: Nem létező könyvtár – Nem létező, vagy nem írható könyvtár megadásakor hibaüzenet jelenik meg.
  • TS-03: Sikeres template mentés – Versions modulban verziók kiválasztása + Template név megadása után a Save as Template gombra a Templates.txt bővül egy új sorral, ami visszaolvasható a Templates modulban.
  • TS-04: Template törlése – Templates modulban jelölt sor törlését követően a Templates.txt fájlból eltűnik a megfelelő sor, a lista frissül.
  • TS-05: Git nélkül létrehozott repository – Git kapcsoló kikapcsolt állapotában a repository létrejön git inicializálás nélkül.
  • TS-06: Git létrehozással – Git kapcsoló bekapcsolva, Git Auth adatok helyesek; repository létrejön, lokálisan git init fut, remote beállítás és első push sikeres.
  • TS-07: IDE megnyitás – IDE kiválasztása (pl. VS Code) esetén a repository sikeres létrehozása után az IDE a megfelelő projekt mappával indul.

10.2. Elfogadási kritériumok

  • A felhasználó a kezdőképernyőről navigálhat a három fő modulra (Versions, Repository Creator, Templates).
  • A felhasználó kiválaszthatja a PHP, Composer és Node.js verziókat.
  • A felhasználó menthet és betölthet sablonokat.
  • A felhasználó elindíthatja a fejlesztői környezetet a kiválasztott beállításokkal.
  • A felhasználó új repository-t hozhat létre érvényes névvel és útvonallal.
  • Hiba esetén figyelmeztető üzenet jelenik meg, amely érthető.
  • A felhasználó szerkeszthet és törölhet sablonokat.

11. Hibakezelés és naplózás

11.1. Hibatípusok

  • Validációs hibák: hiányzó vagy érvénytelen mezők (pl. üres projekt név, invalid verzióformátum).
  • IO hibák: fájl/ könyvtár nem érhető el, nem írható, Templates.txt sérült.
  • Git/GitHub hibák: sikertelen git init, Remote hozzáadás hiba, hitelesítési hiba, push sikertelen.
  • IDE indítási hibák: hiányzó IDE, rossz elérési útvonal, futtatás közben keletkezett kivétel.

11.2. Hibamegjelenítés

  • Felhasználói hibák (validáció) – MessageBox + a releváns mező vizuális kiemelése.
  • Rendszerszintű hibák (IO, Git) – MessageBox rövid, érthető üzenettel

12. Integrációk

12.1. Git (lokális)

  • A repository létrehozás után az alkalmazás futtatja a szükséges git parancsokat (pl. git init, git add ., git commit).
  • Git hiánya esetén a rendszer hibát jelez, és a Git-funkciókat nem próbálja meg erőltetni.

12.2. GitHub (opcionális)

  • A Git Auth popupban megadott felhasználónév és token alapján történik a kapcsolat.
  • Új repository létrehozásakor:
    • GitHub API-n keresztül létrejön egy új tároló.
    • A lokális repository remote-ként megkapja az új GitHub URL-t.
    • Első push végrehajtása (pl. git push -u origin main).
  • Hitelesítési hiba esetén egyértelmű hibaüzenet tájékoztatja a felhasználót (pl. „Érvénytelen token vagy jogosultság hiánya”).

12.3. IDE indítás

  • Az IDE-k támogatása konfiguráció alapján történik (pl. a kiválasztott IDE-hez tartozó futtatható fájl útvonala).
  • Példák:
    • VS Code: code <projectPath>

13. Telepítés és futtatás

13.1. Telepítés

  • .NET Framework 4.7.2 telepítése, ha még nincs jelen a rendszeren.
  • A Yago alkalmazás futtatható állományának kimásolása egy tetszőleges, írható könyvtárba.
  • Opcionális: telepítő készítése (pl. MSI vagy ClickOnce), amely automatikusan létrehozza a szükséges mappastruktúrát (pl. configs).

13.2. Első indítás

  • Az alkalmazás első indításakor, ha nem létezik Templates.txt, üres fájl jön létre.

13.3. Frissítés

  • Az alkalmazás nem tartalmaz automatikus verziófrissítést.
  • Frissítéskor a régi futtatható állományok cseréjére van szükség; a Templates.txt és egyéb konfigurációk megőrizhetők.

14. Kockázatok és jövőbeli fejlesztések

14.1. Kockázatelemzés

  • GitHub API változás: A Git integráció hibássá válhat, ha az API változik vagy a token kezelés módja módosul.
  • Templates.txt sérülése: A sablonok elveszhetnek vagy hibásan töltődhetnek be.
  • Hardkódolt verziólista: Ha a verziók nem konfigurációból érkeznek, a karbantarthatóság romlik (gyakoribb build szükséges verzióváltáskor).
  • Felhasználói hibák: Elégtelen mezővalidáció esetén hibás konfigok jöhetnek létre.

14.2. Jövőbeli funkciók

  • Automatikus frissítés és verzióellenőrzés: Online ellenőrzés és sablonfrissítés.
  • Sablonok felhőalapú szinkronizálása: Több eszköz közötti megosztás.
  • Többfelhasználós támogatás: Bejelentkezési rendszer és profilok.
  • Projektek exportálása és megosztása: Teljes konfiguráció átvitele más felhasználókhoz.
  • Fejlettebb hibakezelés és naplózás: Részletes logfájlok létrehozása, debug mód.
  • Plugin rendszer: Új app típusok és generátorok csatolása pluginekkel.
