Rendszerterv

Projekt neve: Yago Environment Builder

Dátum: 2025.11.07

Készítette: Lénárt János, Ozibiusz Gergő, Köhler Gergő

1. Bevezetés

A Yago Environment Builder egy Windows Forms alapú asztali alkalmazás, amely a fejlesztők munkáját támogatja azáltal, hogy automatizálja és leegyszerűsíti különböző fejlesztői környezetek, illetve repositoryk létrehozását.

Jelen rendszerterv célja, hogy a funkcionális specifikációban meghatározott elvárásokat műszakilag pontosítsa, és bemutassa a megvalósítás architekturális felépítését, adatmodelljét, folyamatait és technikai követelményeit.

2. Rendszer célja és környezete

2.1 Rendszer célja

  • A fejlesztői környezetek (PHP, Node.js, Composer verziók) gyors, ismételhető konfigurálása.
  • Sablonok (templates) segítségével egységes, újrafelhasználható beállítások kezelése.
  • Új projektek és repositoryk automatikus létrehozása a megadott beállítások alapján.

2.2 Rendszer környezete

  • Alkalmazás típusa: Windows Forms asztali alkalmazás
  • Fejlesztési nyelv: C#
  • .NET verzió: .NET Framework 4.7.2 szükséges a futtatáshoz
  • Adattárolás: helyi szöveges fájl (Templates.txt) a sablonok számára
  • Cél platform: Windows operációs rendszer, egyfelhasználós környezet

3. Logikai architektúra

3.1 Rétegek áttekintése

  • Prezentációs réteg (UI): Windows Forms alapú felhasználói felületek (EntryPoint, Versions, Repository Creator, Templates).
  • Üzleti logikai réteg: a verzióválasztás, sablonkezelés, repository generálás működését megvalósító osztályok.
  • Adatkezelési réteg: a sablonok és beállítások fájl alapú tárolásáért felelős komponensek.

3.2 Fő modulok és felelősségek

  • EntryPoint modul:
    • Alkalmazás kezdőképernyője.
    • Navigáció biztosítása a Versions, Repository Creator és Templates modulokhoz.
  • Versions modul (FormVersions):
    • PHP, Node.js, Composer verziók kiválasztása.
    • Környezet indítása.
    • Sablon mentése és betöltése a Templates.txt fájlból.
  • Repository Creator modul (FormRepositoryCreator):
    • App típusának, projekt nevének és mentési helyének megadása.
    • Verziók átvétele a Versions modulból (popup „Select Version”).
    • Repository-struktúra létrehozása, opcionális Git repository készítés, push, IDE megnyitás.
  • Templates modul (FormTemplates):
    • Sablonok listázása táblázatos formában.
    • Sablonok szerkesztése (popup), törlése és mentése.
    • Templates.txt fájl karbantartása.

4. Adatmodell és fájlstruktúra

4.1 Fő adatelemek

  • Sablon (Template):
    • Név (template name).
    • Kapcsolódó PHP, Composer, Node.js verziók.
  • Verziók (Versions):
    • PHP verzió.
    • Composer verzió.
    • Node.js verzió.
  • Repository beállítások:
    • App típusa (React, Laravel, Angular, stb.).
    • Projekt név.
    • Mentési hely (lokális útvonal).
    • IDE típusa (None, VS Code, Phpstorm).
    • Git létrehozása, GitHub autentikációs adatok megléte.

4.2 Templates.txt fájl

  • Tárolás módja: helyi szöveges fájl, soronként egy sablonnal.
  • Adatstruktúra: a sablon mezői előre definiált, konzisztens formában kerülnek sorba (';' elválasztó karakterekkel).
  • Műveletek:
    • Új sablon hozzáadása (Mentés sablonként).
    • Sablon adatainak frissítése (Edit Template → Save).
    • Sablon törlése (Delete Template, megerősítés után).

5. Folyamatok és vezérlés

5.1 Versions modul folyamata

  • A felhasználó megnyitja a Versions fület.
  • A PHP, Composer és Node.js verziókat legördülő listákból kiválasztja.
  • Environment Start gombra kattint.
  • A rendszer ellenőrzi, hogy:
    • minden kötelező mező ki van-e töltve,
    • a verziók formátuma érvényes-e.
  • Sikeres ellenőrzés esetén a rendszer inicializálja a környezeti beállításokat (lokális konfigurációk).
  • A felhasználó megadhat egy template nevet, majd a Save Template gombbal elmentheti az aktuális konfigurációt a Templates.txt fájlba.
  • Meglévő template is betölthető, amely automatikusan beállítja a verziókat.

5.2 Repository Creator modul folyamata

  • A felhasználó megnyitja a Repository Creator fület.
  • Megadja az App típusát, a Projekt nevét és a Mentési helyet.
  • A Select Version gombra kattintva egy popupon kiválasztja a PHP/Composer/Node verziókat (a Versions modul logikája alapján).
  • A Create Repository gombra kattintva a rendszer:
    • ellenőrzi a projekt név és az útvonal érvényességét,
    • létrehozza a mappastruktúrát és az alap fájlokat,
    • ha szükséges, inicializálja a Git repositoryt és első commitot készít,
    • ha be van állítva Git Auth és Git létrehozása, pusholja a GitHubra az új repót,
    • ha az IDE beállítás nem None, megnyitja a repót a kiválasztott IDE-ben.
  • A folyamat végén a felhasználó sikeres vagy sikertelen státuszüzenetet kap.

5.3 Templates modul folyamata

  • A felhasználó megnyitja a Templates fület, ahol táblázatos formában látja a sablonokat.
  • Edit Template: egy sablon kiválasztása után az Edit gombra kattintva popup ablak jelenik meg, ahol a verziók módosíthatók.
  • A Save gomb után a rendszer validálja az adatokat, majd frissíti a Templates.txt fájl megfelelő sorát, és újratölti a listát.
  • Delete Template: a törlendő sablonok checkbox-szal kijelölhetők, majd a Delete Template gomb megnyomására a rendszer megerősítést kér, és jóváhagyás után törli a sorokat a fájlból.

6. Nem funkcionális követelmények

6.1 Teljesítmény

  • A felhasználói felület műveleteire (gombnyomás, váltás) 1 másodpercen belüli reakció szükséges.

6.2 Megbízhatóság

  • Hiba esetén figyelmeztető üzenet jelenik meg a felhasználó számára.

6.3 Használhatóság

  • Egyszerű, jól áttekinthető Windows Forms felületek.
  • Minden fontos funkció maximum néhány kattintással elérhető (EntryPoint → modul → művelet).
  • Hibák és sikeres műveletek esetén egyértelmű visszajelzés.

6.4 Karbantarthatóság és bővíthetőség

  • Külön modulok a verziókezelésre, sablonkezelésre és repository generálásra.
  • Új app típusok vagy további verziók hozzáadása minimális kódmódosítással megoldható.
  • A sablonformátum egyszerű, fájl alapú, így külső eszközökkel is szerkeszthető.

7. Telepítés és üzemeltetés

  • Követelmények:
    • Windows operációs rendszer.
    • .NET Framework 4.7.2
    • Git Command Line Interface (GIT CLI) szükséges az új repository pusholásához.
  • Telepítés:
    • Az alkalmazás futtatható állományának másolása a célgépre.
  • Frissítés:
    • Az alkalmazás nem tartalmaz automatikus frissítési mechanizmust.
    • Frissítéskor a futtatható állomány manuális cseréje szükséges.

8. Kockázatok és jövőbeli bővítési lehetőségek

  • Kockázatok:
    • Külső eszközök (Git, IDE-k) elérhetősége és verziókompatibilitása.
    • Hibás vagy kézzel módosított Templates.txt fájl inkonzisztens beállításokat okozhat.
  • Jövőbeli bővítések:
    • Automatikus frissítés és verzióellenőrzés (online ellenőrzés).
    • Sablonok felhőalapú szinkronizálása több gép között.
    • Többfelhasználós mód, belépés és profilok kezelése.
    • Részletes naplózás és fejlettebb hibanapló.
